# 二分类 CTR 模型加 item_id 算泄露吗？从多分类踩坑到 CT R 的深入思考

## 上篇文章的思考题

之前那篇《多分类模型中物品侧特征的信息泄露》留了个问题：

> "假设你的任务是预测用户是否会点击某个商品（二分类 CTR），你把 `item_id` 加进模型，会泄露信息吗？答案和本文的场景有什么不同？"

收到了不少读者的回复，答案分歧很大：

- 正方："加 item_id 就是作弊，模型直接记住了每个商品的点击率，离线高线上崩"
- 反方："item_id 是 CTR 模型的标准特征，工业界都在用，怎么可能是泄露"

谁是对的？答案是：**都对，但不完全对**。关键在于理解"泄露"在这两个场景中的本质区别。

## 多分类 vs 二分类：泄露的数学本质不同

先回顾一下多分类的泄露场景。

### 场景一：多分类（评分预测）

- 目标：3662 个分类
- 特征：`movie_title`（4268 个唯一值）
- 每个 movie_title 在训练集中几乎**只对应一个评分**

```
movie_title=123  →  rating=5  (100% 的样本)
movie_title=456  →  rating=3  (100% 的样本)
```

这是一个**几乎确定的映射**。用信息论的术语说，特征 $X$ 和目标 $Y$ 之间的互信息 $I(X; Y)$ 接近 $H(Y)$——特征包含了目标的几乎所有信息。

**这就是泄露：模型不需要学习任何泛化知识，只需要查表。**

### 场景二：二分类 CTR

- 目标：{0, 1}（点击/不点击）
- 特征：`item_id`（假设 10 万个商品）
- 每个 item_id 对应**大量正例和负例**

```
item_A  →  click=1 (5% 的样本), click=0 (95% 的样本)
item_B  →  click=1 (0.1% 的样本), click=0 (99.9% 的样本)
```

同一个 item_id 同时产出 click=1 和 click=0 的样本，不存在"一个特征值唯一确定一个目标"的映射。模型学到的其实是 **item 的 base CTR**——这本身是一个合理的先验信号。

**这不是泄露，这是特征提取。** 模型学习的是：

$$P(click=1 | item=A, user=U) \approx \sigma(embedding_{item_A} + embedding_{user_U} + \dots)$$

`item_id` 的 embedding 贡献的是**偏置项**（item 的热门程度），具体是否点击还需要用户特征、上下文特征来共同决定。

## 为什么工业界 CTR 模型普遍用 item_id？

如果你去看看 Google、Meta、字节跳动的 CTR 模型论文，几乎所有模型都使用 item_id/page_id/ad_id 作为特征。原因如下：

1. **归纳偏置**：每个商品有固有的"热门程度"（base CTR），item_id embedding 天然能学到这个偏置。没有 item_id，模型只能通过其他特征间接推测热门程度，效果更差。

2. **不存在一对一映射**：一个商品被展示给成千上万个用户，有的点击有的不点，模型无法靠 item_id 记忆答案。

3. **Embedding 的泛化价值**：Item embedding 在训练过程中会聚类的——相似的商品（价格区间、品类、风格相似）会学到相近的向量，这对冷启动物品也有一定的迁移能力。

## 那么什么时候加 item_id 会出问题？

虽然 item_id 本身不算泄露，但在某些场景下它确实会带来问题：

### 1. 样本切分不合理

```python
# ❌ 错误：随机切分，同一个 item 同时出现在训练和验证集
train, val = random_split(df)

# ✅ 正确：按 item 切分，验证集全是未见过的新品
train = df[df["item_id"].isin(train_items)]
val = df[~df["item_id"].isin(train_items)]
```

如果随机切分，item_id embedding 在验证集上"见过"同样的商品，会高估模型的泛化能力。（这仍然是模型正确学到了 base CTR，只是验证方式不公平。）

### 2. 线上出现大量新商品

如果业务场景是**商品冷启动**（比如电商每天上架数万新品），训练集里的 item_id 在线上大量缺失，模型的表现会明显下降。此时可以考虑：

- 用商品的属性特征（品类、价格、品牌等）代替 item_id
- 保留 item_id，但对 OOV item 的 embedding 做 pooling（聚合同类商品的 embedding）

### 3. Item-level 统计特征构造不当

```python
# ✅ 安全：item 的历史 CTR（用时间窗口截断，不使用未来信息）
item_ctr = df[df["timestamp"] < sample_time].groupby("item_id")["click"].mean()

# ❌ 严重泄露：用样本自身计算 item 的 CTR
item_ctr = df.groupby("item_id")["click"].transform("mean")
```

后者的问题才是真正的泄露——用当前样本的 label 去构造当前样本的特征，与多分类场景的 `user_movie_rate` 是同一类问题。

## 一条判断准则

我总结了一个简单的判断思路，分享给大家：

> **如果某个特征值在训练集中永远只对应同一个 label，它就是泄露。反之则不是。**

- `movie_title=123 → rating=5`（100% 训练样本）→ **泄露**
- `item_A → click=1（5%）, click=0（95%）` → **不是泄露**
- `item_A → click=1（100% 训练样本）` → **是泄露**（但通常不会发生，除非数据集极小或采样有问题）

这条准则对应到信息论上就是：**特征与目标的互信息 $I(X;Y)$ 相对条件熵 $H(Y|X)$ 的大小。** 如果 $H(Y|X) \approx 0$，那就是泄露。

## 两张对照表

| 维度 | 多分类评分预测 | 二分类 CTR |
|------|--------------|-----------|
| 目标空间 | 3662 个类别 | 2 个类别 |
| 特征值-目标映射 | 近似一对一 | 一对多 |
| 模型学到的内容 | 查表映射 | Item base CTR + 高阶交互 |
| 加 item_id 是否泄露 | 是 | 否（但需注意样本切分） |
| 主要风险 | 特征直接暴露目标 | OOV 冷启动 + 不合理的统计特征 |
| 工业界实践 | 不用 item 类特征 | 广泛使用 item_id |

## 一些例外

当然，现实世界总有例外。以下几种情况即使 CTR 模型也要禁用 item_id：

1. **广告点击预测中的广告 ID**：如果训练/测试是按时间切分的，而广告的投放周期很短（如 1 天），训练集的广告在测试集中几乎都不存在，item_id 反而变成噪声。这种情况下一些论文会建议去掉 AD ID。

2. **重复样本问题**：如果数据预处理出错，同一条样本重复出现多次（甚至同一个 (user, item, time) 组合出现正反两个 label），item_id 可能学到错误信号。

3. **用户-物品组合特征**：`user_item_rate`（用户对某个具体物品的评分）这个特征无论是多分类还是二分类都大概率泄露，因为它直接编码了 user 对该 item 的偏好。

## 最后

回到最开始的问题：CTR 中加 item_id 算泄露吗？

**结论：正常情况不算，但不能无脑加。**

- 区分"泄露"和"冷启动退化"两个概念。泄露是不可接受的（模型学会作弊），冷启动退化是可以接受的工程权衡（模型在未见过的 item 上表现变差，但可以其他方式缓解）。
- 判断标准不是"特征看起来是否太强"，而是"特征值与目标值是否构成确定性的映射关系"。
- 如果非要用一个简单规则来判断：把特征分组后统计每个组内 label 的分布，如果大部分组都是纯的（只有一个 label），那就要警惕了；如果分布和整体差不多，那大概率安全。
