# 多分类模型中物品侧特征的信息泄露：离线 AUC 0.98，线上却崩了

## 引子

"离线 AUC 0.98！"

我看着训练日志，心里一阵狂喜。模型在测试集上也跑了 0.97，简直是 SOTA 级别的结果。团队成员都很兴奋，觉得这个版本的模型稳了。

一周后上线，线上 AUC 0.55。

比随机好一点，但基本等于废了。更诡异的是，同样的模型结构、同样的超参数，之前跑其他数据集都好好的，为什么这次翻车了？

排查了一整天，最终发现罪魁祸首是**特征工程中一个看似无害的选择**——我们把电影标题（movie_title）加进了模型。

## 问题现象

先交代一下背景。这是一个电影评分预测任务，多分类，目标是把 3662 部电影按评分分类。我们用的模型叫 GwEN（Group-wise Embedding Network），是一种处理稀疏特征的深度模型。

离线指标好得离谱。训练集 loss 迅速收敛到接近 0，验证集 AUC 稳定在 0.97+，测试集也差不多。按经验来说，这种泛化性能不太对劲，但当时谁也没多想——毕竟特征工程做了很多，效果好似乎也合理。

直到线上数据回流，才发现真实场景的预测效果一塌糊涂。最明显的特征是：**大量预测结果集中在少数几个类别上，跟随机猜测几乎没区别。**

## 原因分析

逐层排查，最终定位到特征配置。

我们的多分类模型用了 50+ 个特征，其中一部分是物品（电影）侧特征，包括：

| 特征名 | 唯一值数 | 说明 |
|-------|---------|------|
| `movie_title` | 4268 | 电影标题 ID |
| `movie_genres` | 19 | 电影类型 |
| `user_movie_rate` | 3569 | 用户对某部电影的历史评分 |

目标分类数：**3662**。

看出问题了吗？**movie_title 的唯一值数（4268）几乎等于目标分类数（3662）**。模型只要记住 `movie_title_123 → class_456` 这个映射表就能拿到极低的 loss。它根本不需要学习用户行为、历史偏好、上下文信息——那些复杂的特征对它来说只是噪声。

更隐蔽的泄露来自 `user_movie_rate`。这个特征本身就是"用户对某部电影的评分"，而我们要预测的也是"用户对某部电影的评分"。训练集中如果存在同一对 (user, movie) 在训练和验证样本中，验证集的 AUC 也一起虚高了。

这种泄露在排行榜竞赛中有一个专门的术语叫 **label leakage**——预测目标的信息被间接或直接地包含在输入特征中。

### 为什么离线 AUC 高但线上崩？

用数学语言描述：设真实评分 $y$，泄露特征 $x_{leak}$，泛化特征 $x_{gen}$。

训练时模型学到的近似函数是：

$$\hat{y} = f(x_{leak}, x_{gen}) \approx f(x_{leak})$$

因为 $x_{leak}$ 提供了 $y$ 的几乎全部信息，模型自动**忽略**了 $x_{gen}$ 的梯度信号。

线上推理时，$x_{leak}$ 对应的特征值（新电影 ID、新的 user-movie 组合）在训练集中没有出现过，embedding 层 fallback 到随机初始化的向量：

$$\hat{y}_{online} = f(\text{OOV}, x_{gen}) \approx \text{random}$$

因为 $f$ 从未学会利用 $x_{gen}$，结果退化到随机猜测。

### 问题本质

这不是模型的问题，是**特征工程中的因果倒置**。我们做的事情等价于用一个学生是否知道考试答案来预测他的成绩——如果"知道答案"这个特征存在，模型预测会 100% 准确，但完全无法推广到新考试。

## 如何解决

### 第一步：识别泄露特征

最直接的筛查方法是检查特征的 Unique 值与目标分类数的关系：

```python
import pandas as pd

# 粗略筛查：基数接近目标分类数的特征
target_classes = df["rating"].nunique()
for col in df.columns:
    n_unique = df[col].nunique()
    ratio = n_unique / target_classes
    if ratio > 0.3:   # 超过目标分类数 30% 就有嫌疑
        print(f"[WARN] {col}: {n_unique} / {target_classes} = {ratio:.2%}")
```

但这只是一个粗筛，更准确的判断需要检查**特征值与目标值的映射关系**：

```python
# 精确判断：特征值是否几乎唯一映射到一个目标
for col in candidate_cols:
    mapping = df.groupby(col)["rating"].nunique()
    one_to_one_ratio = (mapping == 1).mean()
    print(f"{col}: 一对一映射比例 = {one_to_one_ratio:.2%}")
```

如果某个特征的几乎所有值都只映射到一个分类（比如 movie_title 每条记录对应一个固定评分），那就是确凿的泄露。

### 第二步：分类处置

我把泄露特征分为三类，处理方式不同：

| 类型 | 特征举例 | 基数 | 处理方式 |
|------|---------|------|---------|
| **严重泄露**（一对一映射） | `movie_title`, `user_movie_rate*` | ≈目标分类数 | **直接禁用** |
| **间接泄露**（User-Item 交叉统计） | `user_genres_rate*` | 固定（如 19） | **禁用**或改为统计量 |
| **特征退化**（区分度太低） | `item_id_idx` 等纯索引 | 取决于切割方式 | **禁用** |

对于我们的场景，最终禁用了 18 个特征：

- **movie_title**（vocab_size=4268）：电影与目标一一对应
- **user_movie_rate 系列**（vocab_size≈3558~3569）：User-Item 评分直接暴露目标
- **user_genres_rate 系列**（vocab_size=19）：用户对电影类型的偏好间接泄漏评分信息

### 第三步：保留物品侧信息的安全做法

完全去掉物品侧特征可能导致模型信息不足。安全的做法是**做聚合统计**：

```python
# ❌ 不安全：直接用 item_id
item_id  → embedding  → concat

# ✅ 安全：物品侧聚合统计量
item_avg_rating    # 物品的平均评分（全量统计）
item_rate_count    # 物品的评分次数
item_popularity_rank  # 物品的热度排名
```

关键原则：**不要使用样本自身的目标值参与构造该样本的特征**。任何以 (user, item) 为粒度、直接涉及评分的字段都要格外警惕。

### 第四步：交叉验证确认

改完特征后，做一次严格的**时间序列切分验证**：

```python
# 按时间切分，防止未来信息泄漏到过去
train = df[df["timestamp"] < split_time]
val = df[(df["timestamp"] >= split_time) & (df["timestamp"] < test_time)]
test = df[df["timestamp"] >= test_time]
```

如果时间切分下的 AUC 显著低于随机切分，说明数据中仍然存在泄露。

## 最后

这次踩坑让我想起 Andrej Karpathy 的一句话：

> "A model can only learn what you show it, but it will learn exactly what you accidentally leak."

模型学到的东西比你以为的要多——尤其是你"不小心"泄露给它的信息。特征工程不只是选好特征，更重要的是**剔除那些看起来有用、但实际上在作弊的特征**。

这几个特征不禁用会带来不同严重程度的问题：

1. movie_title（vocab_size=4268, 目标=3662）— 严重泄露
电影标题几乎与电影一一对应。模型只需记住 movie_title_123 → class_456，不用学任何用户行为信号，离线准确率会极高，但线上遇到没见过的电影就直接失效。
2. user_movie_rate* 系列（vocab_size≈3558~3569）— 直接泄露标签
特征值本身就是"用户对某部电影的评分"，而目标也是"预测用户对某部电影的评分"。这相当于把答案直接放在输入里，模型学到的是恒等映射，完全没有泛化能力。训练集和验证集是随机切分的，如果同一对 (user, movie) 同时出现在训练和验证中，验证指标也会虚高。
3. user_genres_rate* / user_genres_rate_cnt* / user_top3_genres（vocab_size=19）— 中度泄露
基数 19 不高，但问题是：这些特征编码的是"用户对某个电影类型的评分"，而目标电影属于特定类型。如果某个用户只对"动作片"打过高分，模型可以推断该用户对任何动作片都会打高分——这看起来合理，但问题是这个特征已经编码了用户对目标电影的评分信息，属于间接泄露。训练集中用户对动作片 A 的偏好会直接泄露到测试集的动作片 B 上。
4. movie_genres（vocab_size=19）— 理论上安全，可能误禁
19 个类型对 3662 个电影，不存在一对一映射，理论上是一个合理特征。可能被禁的原因：
- MovieLens 数据中每部电影只有一个类型，区分度太低
- 或者只是之前测试时顺手禁掉，实际影响不大

上线后模型表现会断崖式下跌，具体场景不同：
带泄露特征的模型：
- 离线 AUC 0.98，线上 0.55
- 线上请求的特征值（新电影 ID、新 user_movie_rate）在训练集中没见过，模型 fallback 到默认 embedding，输出基本是随机猜测
不带泄露特征的模型：
- 离线 AUC 0.82，线上 0.80
- 差距来自分布偏移（时间、用户群体变化等），属于正常衰减
核心原因：泄露特征在训练时是强判别信号，模型几乎完全依赖它来预测，忽略了真正的泛化特征（用户历史行为、画像等）。上线后特征值变为 OOV（Out-Of-Vocabulary），模型失去主要信号源，表现退化到随机水平。
